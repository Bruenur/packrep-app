# AsyncStorage → Supabase Migration Guide (Phase 1)

Goal

Preserve the existing AsyncStorage-based flows while adding Supabase as a cloud-backed option. Migration should be incremental, reversible, and non-destructive.

High-level strategy

1. Introduce Supabase tables (see `supabase/schema.sql`) that mirror the shape of data currently stored in AsyncStorage.
2. Add migration utilities that read AsyncStorage keys and upsert corresponding rows into Supabase. Track migration status per key/item with a small migration tracking table.
3. Use a read-through strategy: keep existing reads from AsyncStorage as the source of truth until a successful cloud sync exists. New reads may optionally prefer cloud if available.
4. Implement conflict resolution rules: last-write-wins by timestamp for Phase 1; record provenance in audit fields.

Suggested incremental steps

- Step 0 (safety): Add backups/export tooling for AsyncStorage before running any migration.
- Step 1: Add the Supabase client wrapper (`src/lib/supabase.ts`) and API helpers (`src/lib/api.ts`). Do not wire into screens.
- Step 2: Create migration scripts that:
  - iterate AsyncStorage keys of interest,
  - read the stored JSON payload,
  - upsert into Supabase tables with an `original_key` or `source` column,
  - insert an entry in `migrations_asyncstorage` to mark the key/item migrated.
- Step 3: Run migration in a development environment and verify data integrity in Supabase dashboard.
- Step 4: Add optional background sync that periodically reconciles AsyncStorage → Supabase and Supabase → AsyncStorage deltas.

Mapping guidance

- Map simple objects to a single table row and complex objects to a JSONB `payload` column if the schema is uncertain.
- Example mapping:
  - AsyncStorage key `user_profile` → `users` table (columns: `id`, `email`, `profile jsonb`)
  - AsyncStorage key `leads_*` → `leads` table (columns: `id`, `owner_id`, `payload jsonb`)

Conflict resolution (Phase 1)

- Prefer deterministic, simple rules for now:
  - If remote row timestamp > local timestamp, overwrite local.
  - Else, push local to remote (upsert).
- Record `migrated_at` and `source` fields so future logic can be more sophisticated.

Rollback plan

- Keep local AsyncStorage data until you’ve verified Supabase rows are complete. If an issue occurs, clear the `migrations_asyncstorage` table to allow re-run, or use exported AsyncStorage backups to restore state.

Testing

- Test migrations on a small dataset first. Verify each migrated item in Supabase and the `migrations_asyncstorage` table.

Migration tooling (scaffold)

Files added:

- `src/lib/migrations.ts` — library with mapping from AsyncStorage keys to Supabase tables, helpers for counting local records, validating required fields, preparing upsert payloads, recording migration status, and a `runMigration()` entrypoint. It supports dry-run mode and will not write to the DB when dry-run is enabled.
- `scripts/migrate_asyncstorage.ts` — CLI scaffold to run migrations (supports `--dry-run`/`-d` and optional list of keys to migrate). Run with `ts-node` or compile first.

Key → Table mapping (implemented):

- `PR_leads_v8_hybrid` → `leads`
- `PR_builders_v8_inventory` → `builders`
- `PR_used_homes_v7_inventory` → `used_homes`
- `PR_pins_v7_inventory` → `favorite_pins`
- `PR_profile_v8_hybrid` → `users`
- `PR_appointments_v1` → `appointments`
- `PR_pin_prefs_v1` → `pin_prefs`
- `PR_wall_posts_v1` → `wall_posts`
- `PR_open_house_requests_v1` → `open_house_requests`

Usage (dry-run recommended):

1. Review `supabase/schema.sql` in the Supabase project (apply schema in staging).
2. From the repo root, run the migration script in dry-run mode to preview actions:

```bash
# with ts-node
ts-node scripts/migrate_asyncstorage.ts --dry-run

# or compile and run later; pass specific keys to limit scope
ts-node scripts/migrate_asyncstorage.ts --dry-run PR_leads_v8_hybrid
```

3. Inspect the dry-run output logs to confirm how many items would be inserted and any missing required fields.
4. When confident, run without `--dry-run` to perform upserts to Supabase:

```bash
ts-node scripts/migrate_asyncstorage.ts
```

Notes & caveats

- The scaffold writes summary rows into `migrations_asyncstorage` unless `--dry-run` is used. The `migrations_asyncstorage` table is intended to track which AsyncStorage keys have been migrated and basic telemetry.
- Local item IDs generated by the app may not be valid UUIDs expected by the Supabase schema's `uuid` PK columns. The scaffold conservatively inserts rows using explicit columns when safe and otherwise stores original objects in `payload jsonb` to avoid UUID validation failures. Expect to refine the DDL or add an `orig_id` column if you want 1:1 id preservation.
- This scaffold intentionally does not modify `src/lib/storage.ts` or UI screens; it reads via the existing loader functions so local behavior remains untouched.

Important: Supabase auth / user mapping required before real migration

- Real migration of user-scoped records (favorites, pin_prefs, appointments, open-house requests, etc.) requires mapping local ownership to Supabase `users.id` values. The migration scaffold supports passing an `ownerId` and `requesterId` (CLI flags `--owner-id` and `--requester-id`) to attach ownership during upsert. Without those values, the script will warn in dry-run and will skip inserting `pin_prefs` during a real run.
- Recommended flow: provision Supabase users (via sign-up or admin mapping), export a mapping of local identity → Supabase `users.id`, then run migration providing `--owner-id`/`--requester-id` (or implement per-item owner mapping in the migration code).
